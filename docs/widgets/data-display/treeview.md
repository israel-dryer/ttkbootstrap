# TreeView

`TreeView` is a themed wrapper over `tkinter.ttk.Treeview` that
displays **hierarchical or flat tabular records** as expandable
rows. It is the underlying control [TableView](tableview.md) is
built on, exposed directly when the data has parent/child structure
(folder trees, outlines, category navigation) or when you need raw
tree-with-columns rendering without the toolbar, search, paging,
and SQL data source TableView adds.

The widget keeps the entire `ttk.Treeview` API â€” `insert`,
`delete`, `move`, `item`, `column`, `heading`, `selection_*`,
`tag_configure`, `xview` / `yview`, and so on â€” and adds theme
tokens, a custom border element, density-aware row heights, and
header / selection background overrides.

## Basic usage

```python
import ttkbootstrap as ttk

app = ttk.App()

tv = ttk.TreeView(app, columns=("size",), show="tree headings")
tv.heading("#0", text="Name")
tv.heading("size", text="Size")
tv.column("size", width=100, anchor="e")

src = tv.insert("", "end", text="src", open=True)
tv.insert(src, "end", text="app.py", values=("4.2 KB",))
tv.insert(src, "end", text="utils.py", values=("1.8 KB",))
tv.insert("", "end", text="README.md", values=("2.0 KB",))

tv.pack(fill="both", expand=True, padx=20, pady=20)

app.mainloop()
```

`Treeview` (lowercase) is kept as a back-compat alias for the
ttk-style spelling â€” `ttk.Treeview(...)` and `ttk.TreeView(...)`
construct the same class.

## Data model

TreeView holds its data internally as a forest of items keyed by
**iid**. Every `insert(...)` call returns an iid (autogenerated if
you don't pass one); every other method addresses items by that
iid.

| Concept       | Meaning                                                                                          |
| ------------- | ------------------------------------------------------------------------------------------------ |
| `iid`         | Unique string identifier for an item. Autogenerated as `I001`, `I002`, â€¦ if not supplied.        |
| `parent`      | The iid of the containing item, or `""` for top-level rows.                                      |
| `text`        | The string shown in the tree column (`#0`).                                                      |
| `values`      | A tuple/list of cell values, one per `columns=` entry. Indexed by column id.                     |
| `open`        | Whether this row's children are visible (`True`/`False`).                                        |
| `tags`        | List of tag strings used for per-row styling and event filtering.                                |
| `image`       | A `PhotoImage` placed before `text` in the row.                                                  |

### Display modes

The `show` option selects which parts render:

| `show=`            | Tree column (`#0`) | `columns=` headings | Use for                                  |
| ------------------ | ------------------ | ------------------- | ---------------------------------------- |
| `"tree headings"`  | Yes                | Yes                 | Default â€” folder-with-detail layouts.    |
| `"tree"`           | Yes                | No                  | Pure outline / hierarchy navigators.     |
| `"headings"`       | No                 | Yes                 | Flat tabular display (no nesting).       |
| `""`               | No                 | No                  | Rare; canvas-only rendering.             |

### Inserting and mutating

```python
# Insert under a parent iid; "" places the row at the top level
folder = tv.insert("", "end", text="docs", open=True)
tv.insert(folder, "end", text="README.md", values=("2.0 KB",))

# Custom iid for stable references
tv.insert("", "end", iid="root", text="Project")
tv.insert("root", 0, text="config.json")

# Update fields via `item(iid, **opts)`
tv.item("root", open=False, values=())

# Per-cell write via `set(iid, column, value)`
tv.set("root", "size", "â€”")

# Move and delete
tv.move(folder, "", 0)         # move to first under root
tv.delete(folder)              # also deletes descendants
```

There is **no datasource binding** and **no `signal=` channel** â€”
mutation goes through the ttk API directly. For external state
that needs to drive the tree, watch the source and call
`insert` / `item` / `delete` from your own code.

### Tags

Tags let you apply per-row styling without subclassing the widget:

```python
tv.tag_configure("warning", foreground="#a93232", font="-weight bold")
tv.insert("", "end", text="Quota exceeded", tags=("warning",))
```

`tag_bind` attaches event handlers that fire only on tagged rows:

```python
tv.tag_bind("warning", "<Double-1>", lambda e: print("warning row clicked"))
```

## Common options

| Option              | Purpose                                                                                                  |
| ------------------- | -------------------------------------------------------------------------------------------------------- |
| `columns`           | Sequence of column identifiers (used as the `values` keys and as defaults for headings).                 |
| `displaycolumns`    | Subset and order of `columns` to display. `"#all"` shows them all in declared order.                     |
| `show`              | Which parts to render: `"tree headings"` (default), `"tree"`, `"headings"`, or `""`.                     |
| `selectmode`        | `"browse"` (default; one row), `"extended"` (multi via Ctrl/Shift), or `"none"`.                         |
| `height`            | Visible row count (not pixels). The widget scrolls internally past this.                                 |
| `padding`           | Internal padding around the tree area.                                                                   |
| `surface`           | Surface token used for the body background (default `"content"`).                                        |
| `select_background` | Color token for the selected-row background (default `"primary"`).                                       |
| `header_background` | Color token for column headings (default: an elevated shade of `surface`).                               |
| `border_color`      | Color token for the outer border (default: theme border color over `surface`).                           |
| `show_border`       | Whether to draw the outer border (default `True`). When `False`, the border collapses into the surface.  |
| `density`           | `"default"` or `"compact"`. Compact reduces row height, padding, and indicator size.                     |
| `cursor`            | Mouse cursor over the widget.                                                                            |
| `style`             | Explicit ttk style name (rarely needed; the builder picks one).                                          |

Heading and column geometry is configured **after** construction:

```python
tv = ttk.TreeView(
    app,
    columns=("status", "owner"),
    show="tree headings",
    selectmode="extended",
    select_background="info",
    density="compact",
)

tv.heading("#0", text="Item", anchor="w")
tv.heading("status", text="Status", anchor="w")
tv.heading("owner",  text="Owner",  anchor="w")

tv.column("#0",     width=240, stretch=True)
tv.column("status", width=120, anchor="w")
tv.column("owner",  width=120, anchor="w")
```

`accent` is **not** consumed by TreeView's style builder â€” to
recolor the selected row, the headings, or the border, use
`select_background`, `header_background`, and `border_color`
respectively.

## Behavior

**Selection.** `selectmode="browse"` (default) gives a single
row at a time; clicking a different row replaces the selection.
`"extended"` enables Ctrl-click toggling and Shift-click range
extension. `"none"` displays rows without an active selection
(useful for read-only outline views).

**Keyboard navigation.** When the widget has focus, Up/Down move
between visible siblings, Right opens (or descends into) the
current item, Left collapses (or ascends), and Home/End jump to
the first/last visible row. PageUp/PageDown move by the visible
window.

**Click semantics.** Left-click selects (or extends in
`extended` mode). Click on the indicator chevron toggles
expand/collapse â€” the row is not selected by that click. Double
clicking a parent row also toggles expansion. Right-click does
nothing built in; bind `<Button-3>` (or `<Button-2>` on macOS) to
open a [ContextMenu](../actions/contextmenu.md).

**Scrolling.** TreeView shows scrollbars only when the parent
provides them â€” there is no built-in scrollbar (unlike
[TableView](tableview.md)). Wire `xscrollcommand` /
`yscrollcommand` to a `ttk.Scrollbar`, or place the widget inside a
[ScrollView](../layout/scrollview.md):

```python
sv = ttk.ScrollView(app)
sv.pack(fill="both", expand=True)
tv = ttk.TreeView(sv, columns=("size",))
tv.pack(fill="both", expand=True)
```

**Open/close state.** Set `open=True` at insert time to start a
row expanded. Use `tv.item(iid, open=True)` afterwards. There is no
"expand all" â€” walk `tv.get_children(iid)` recursively if you need
that.

**Indicator rendering.** Tk 8.6.13 introduced a regression in the
custom-image indicator path, so the framework now uses the native
indicator triangle and reserves `open_icon` / `close_icon` only as
fallbacks on older Tk versions. The triangles inherit the
foreground color of the row.

**Density.** `density="compact"` switches body and heading fonts
to the `caption` size, drops row height by ~10 %, tightens cell
padding, and shrinks the expand/collapse indicator. Use it for
data-dense panels where information density matters more than
breathing room.

## Events

| Event               | When                                                  |
| ------------------- | ----------------------------------------------------- |
| `<<TreeviewSelect>>` | The selection changed.                                |
| `<<TreeviewOpen>>`   | An item was expanded (children made visible).         |
| `<<TreeviewClose>>`  | An item was collapsed.                                |

These are emitted by ttk itself. The framework does not add
`on_*` helpers, virtual click events, or insert/update/delete
events on TreeView â€” for that surface, use
[TableView](tableview.md), which wraps TreeView and emits
`<<RowClick>>`, `<<RowInsert>>`, `<<SelectionChange>>`, etc.

The selection at the time of the event is read from the widget,
not from `event.data`:

```python
def on_select(event):
    iids = tv.selection()             # tuple of selected iids
    rows = [tv.item(i) for i in iids]
    print(rows)

tv.bind("<<TreeviewSelect>>", on_select)
tv.bind("<<TreeviewOpen>>",   lambda e: print("opened:",  tv.focus()))
tv.bind("<<TreeviewClose>>",  lambda e: print("closed:",  tv.focus()))
```

For raw click handling that the virtual events don't cover
(double-click, right-click, click on a specific row before
selection settles), bind to the underlying mouse events and
identify the target with `tv.identify_row(event.y)`:

```python
def on_double(event):
    iid = tv.identify_row(event.y)
    if iid:
        print("dbl:", iid, tv.item(iid))

tv.bind("<Double-1>", on_double)
```

## Performance guidance

TreeView is **not virtualized** â€” every `insert(...)` creates a
real Tk item, regardless of whether the row is currently visible.
For modest hierarchies (folders, settings trees, a few hundred
rows), this is fine. For larger datasets, consider:

- **Lazy expansion.** Insert top-level items only; populate
  children on `<<TreeviewOpen>>` and clear them again on
  `<<TreeviewClose>>`. This keeps the live item count proportional
  to what the user has actually opened.
- **Switch to TableView.** For flat, multi-column data past a few
  hundred rows, [TableView](tableview.md) adds paging, virtual
  scrolling, search, and filtering on top of TreeView and avoids
  realizing every row.
- **Batch inserts under one parent.** If you must insert many rows
  at once, pre-build all the rows under a single off-screen parent
  iid first, then `move` the parent into place. Tk redraws the
  region once instead of once per insert.
- **`detach`/`reattach` over `delete`.** When rows reappear after
  a filter, `tv.detach(iid)` removes them from the visible
  hierarchy without destroying them; `tv.reattach(iid, parent,
  index)` restores them in place. Re-inserting destroyed rows is
  significantly more expensive.

## When should I use TreeView?

Use `TreeView` when:

- the data is naturally **hierarchical** (folder trees, outlines,
  category browsers, parent/child relationships)
- you want raw ttk-style row rendering without TableView's
  toolbar, paging, search, or SQL-backed datasource
- you need fine-grained control over per-row styling via tags or
  per-cell writes via `set(iid, column, value)`

Prefer:

- [TableView](tableview.md) â€” for multi-column data that benefits
  from sorting, filtering, paging, search, and built-in editing
- [ListView](listview.md) â€” for flat lists where each record is
  best shown as a rich row (icon, title, body) rather than a row
  of cells
- [Label](label.md) â€” when the content is a handful of static
  rows and a tree control is overkill
